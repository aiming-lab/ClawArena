# HBench Trace Data — 总体设计文档

> **状态**：v0.2（场景 1 设计已迁入 scenario1/）
> **目标**：为 hbench trace benchmark 定义跨场景的设计框架，具体场景实现见 `scenario*/` 子目录。

---

## 1. 设计理念

### 1.1 核心主题：Human-in-the-Loop

强调真实人机协作场景中 agent 的综合能力，下分三个维度：

| 维度 | 含义 | 在数据中的体现 |
|------|------|---------------|
| **多信息源** | 矛盾、残缺、互补的信息分布于不同 session 和 workspace 文件 | 多个会话窗口（DM、群聊）+ workspace 文档，信息交叉验证 |
| **动态** | workspace 和 session 随 update 轮次演化，agent 需追踪变化 | 多次 update，每次引入新 session 消息和/或 workspace 文件 |
| **人类偏好** | agent 需学习并遵守用户的特定工作偏好，即使后续不再提示 | 前期通过 feedback/问题注入偏好规则，后期静默考察是否遵守 |

### 1.2 与现有 trace-bench 的关键差异

| 方面 | 现有 trace-bench | 本设计 |
|------|-----------------|--------|
| 问题类型 | 仅 multi_choice | multi_choice + exec_check |
| Session 内容 | user/assistant 纯对话为主 | 大量 tool call（read, write, exec, search）模拟真实 agent 行为 |
| 任务形式 | 选择题（信息回溯） | 问答 + 文件写入 + 代码执行 + 信息提取 |
| 人类偏好 | 无 | 渐进式偏好注入与静默考察 |
| 规模 | ~228k tokens | 目标 700-800k tokens（最低 300k） |

## 2. 问题类型设计

详见 [Question Types 设计规格](../question-types.md)。

两种类型：
- **multi_choice** — 考察 agent 认知（信息判断、跨源推理、矛盾识别），从回复中提取 `\bbox{}` 答案
- **exec_check** — 考察 agent 行为（文件写入、代码执行、格式合规），运行 shell command 检查副作用

### 题型与维度的关系

| 题型 | 多信息源 | 动态 | 人类偏好 |
|------|---------|------|---------|
| multi_choice | 跨源信息判断、矛盾识别 | 信息更新后答案反转 | 偏好认知考察 |
| exec_check | — | 适应新数据/schema | 格式/命名/代码风格偏好验证 |

## 3. 人类偏好机制设计

### 3.1 偏好注入策略

偏好规则通过四种渠道引入，由显式到隐式递进：

| 阶段 | 注入方式 | 特征 |
|------|---------|------|
| **显式注入** | 直接在 question 中给出规则 | Agent 明确知道规则，但不因违规扣分（用 `pref` 字段） |
| **Feedback 注入** | 上一轮 feedback 中指出偏好违反 | Agent 通过纠正性 feedback 学习规则 |
| **Session 隐式注入** | workspace style guide 或历史 session 中用户纠正 | Agent 需从上下文推断偏好 |
| **静默考察** | 不再提示，直接考察 agent 是否遵守 | 违规直接影响得分（进入 eval.command） |

### 3.2 通用偏好类别

每个场景可选用以下偏好类别的子集（推荐 4-5 条）：

| 类别 | 示例规则 |
|------|---------|
| 输出格式 | 时间戳格式、数字格式、单位规范 |
| 文件命名 | 命名模式（日期前缀、版本后缀、大小写） |
| 文档结构 | 必须包含的节标题、列表格式规范 |
| 代码风格 | docstring、type hints、日志 vs print |
| 沟通习惯 | 总结句、不确定标注、来源引用 |

### 3.3 偏好在题目中的评估

- **显式阶段**：`exec_check` 使用 `pref` 字段独立运行偏好检查，结果不影响得分，只影响 feedback
- **静默阶段**：偏好检查并入 `eval.command`（模式 E 或 F），违规直接扣分
- **multi_choice**：部分选项设计为"符合偏好但信息错误"或"信息正确但违反偏好"的陷阱

## 4. 信息设计原则

### 4.1 多信息源设计模式

| 模式 | 描述 | 每场景目标数量 |
|------|------|-------------|
| **矛盾** | 两个 session/文件对同一事实给出不同说法 | 6-8 组 |
| **残缺** | 单一 session 仅包含部分信息，需跨源拼接 | 5-7 组 |
| **互补** | 多个来源各提供不同角度，综合后得到完整图景 | 4-6 组 |

### 4.2 动态设计模式

| 模式 | 描述 | 每场景目标数量 |
|------|------|-------------|
| **信息反转** | 新 update 推翻之前的结论 | 3-4 个 |
| **渐进明确** | 早期模糊信息随 update 逐步清晰 | 3-4 个 |
| **级联影响** | 一个变更导致多个下游结论需修改 | 2-3 个 |

### 4.3 Session 真实性设计

| 行为类型 | 目标占比（agent 回复） | 示例 |
|---------|---------------------|------|
| **工具调用** | ~35-40% | read 文件、write 文件、exec 命令、search 内容 |
| **多步推理** | ~15-20% | thinking block + 多次 tool call + 最终回复 |
| **纯对话** | ~30-35% | 讨论、确认、提问 |
| **错误/重试** | ~5-10% | tool call 失败、修正后重试 |

## 5. 噪声设计原则

- **Session 噪声**：各 DM/群聊中穿插与项目无关的闲聊，分布不均匀
- **Workspace 噪声**：2-3 个近信号干扰文件（看似相关实为误导），1-2 个纯无关文件
- **噪声约束**：不引入 contradiction map 之外的新矛盾；不在噪声中植入关键信号词

### Token 预算框架

| 组件 | 建议占比 |
|------|---------|
| 有效信号（session + workspace 信息） | ~65-70% |
| 噪声内容 | ~15-20% |
| 系统文件（AGENTS.md 等） | ~2-3% |
| eval/scripts | ~5-8% |

## 6. 场景规格标准

每个场景应包含以下层级文件（参考 scenario1/ 结构）：

| 文件 | 内容 |
|------|------|
| `GUIDE.md` | 场景概述、执行顺序、session 中间格式规格 |
| `layer0-narrative.md` | 时间线、角色、矛盾 map、agent 偏见、偏好注入弧 |
| `layer1-workspace.md` | 所有 workspace 文件的路径、内容要点、加入时机 |
| `layer2-sessions.md` | 所有 session 各阶段的对话段落设计 |
| `layer3-eval.md` | 所有题目（含选项/答案/命令/pref 字段）的完整设计 |
| `layer4-dynamic.md` | 各 update 的触发时机、source 文件清单、action 规格 |
| `layer5-noise.md` | 噪声分布、干扰文件设计、token 分配 |

### 规模基准（每场景）

| 指标 | 建议范围 |
|------|---------|
| 题目数 | 50-70 题 |
| Update 轮次 | 3-5 次 |
| Session 数量 | 5-9 个（DM + 群聊混合） |
| 矛盾数量 | 6-10 条 |
| 偏好规则数 | 4-6 条 |
| 总 token | 500k-900k |
