# EC 题目难度升级思路指南

> 本文记录从 v1（过简）到 v2/v3（适当难度）的核心升级思路，供后续造数参考。
> 对照文档：`plan/v1-plan.md`、`plan/v2-plan.md`、`plan/v3-plan.md`。

---

## 一、v1 被判定为"太简单"的根本诊断

v1 EC 题的本质是**"抄写 + 整理"**：agent 读源文件，把关键数字写进 Markdown，check 脚本只验证字符串是否出现。典型代表：

```python
# v1 check 脚本的实质
"2.3" in content        # ✓
"0.8" in content        # ✓
"1.5" in content        # ✓  (2.3 - 0.8，直接出现在文档里)
len(headings) >= 2      # ✓
→ PASSED
```

Agent 不需要**推理**，只需**搜索 + 粘贴**。对现代大模型而言，这是零难度任务。即便是"写结论报告"这类题，若结论内容不包含矛盾中推理的环节，agent 同样可以用空洞文本糊弄过去。

**关键原则**：**所有题目都必须至少考察某种推理能力，决不能是简单复制粘贴。**

---

## 二、关于 workspace 文件结构

造题时须尊重 workspace 的**现有文件结构**，不强制要求"纯 .md"。每个 workspace 是独立设计的，可能包含 .md 文档、.json 数据、.py 脚本等。

**正确姿势**：造题前先 `ls` workspace 目录，核实实际存在哪些文件，依据实际文件内容出题，而非预设 workspace 的格式。L3 任务（要求 agent 从零创建脚本）是出题设计，不是 workspace 约束——若 workspace 里原本就有某个脚本，可以设计"修改并扩展"类任务；若没有，则设计"从零创建"类任务。

## 三、四条提高难度的路径

### 路径 A：计算派生值（最直接，优先使用）

要求 agent 计算 workspace 文件中**不直接出现**的中间量，check 脚本做数值精度验证。

- **不要**："写一个文件说明文件大小差异"
- **改为**："计算 email 附件大小与官方版本大小的字节级偏差百分比（精确到 0.1%），以及时间差 Δt（精确到秒），写入 JSON 字段"
- **check**：`abs(val - ground_truth) < 0.5`，而非字符串包含

即便 agent 知道目标数字，**写错计算逻辑**也会被精准捕获。

### 路径 B：矛盾裁决（考察判断力）

在 workspace 中故意埋入两条互相矛盾的说法，要求 agent **选定一个来源并说明理由**，check 验证 agent 选择了正确来源且未混用错误来源。

- 例：两个文件中各有不同的"官方播放量"，agent 须判断哪个更可信并给出依据
- check 中加入**负向断言**（M6）：验证 agent 未把错误来源的数值写入结论

### 路径 C：可执行脚本（L3 扩展，与 A 配合效果最强）

要求 agent 从零编写 Python 脚本，读取 workspace 文件并输出 JSON，check 运行该脚本验证字段值精度。

- 原来若干 L2 "写 Markdown" 题，可升为 L3 "写脚本输出 JSON" 题
- 脚本必须**解析**文档内容（而非硬编码数字），才能在 check 中通过
- 典型形式：`python scripts/analyze_xxx.py → stdout JSON → check 验数值`

**路径 A + C 结合效果最强**，是 v2/v3 的主要升级手段。

### 路径 D：严格 schema 验证（JSON 类题升级）

将"字段存在性"验证升级为：
- `strength` 必须是枚举值 `["high", "medium", "low"]`
- 数值 ratio 字段须在合理范围内（`abs(x - expected) <= tol`）
- JSON 数组须按时间序 / 重要性序排列

## 四、题目合并策略：同批次 update 区间内合题

**v1 的问题**：同一 update 区间内堆叠多道 EC，每道只考察一个小点，形成"碎片化"验证。

**v2 的策略**：将同批次 update 之间的多道 EC 题**合并为一道多产物任务**，check 用 `&&` 串联多个验证命令：

```bash
# 合并后的 eval.command 示例
python check_report.py ${workspace} &&
python check_json_schema.py ${workspace}/docs/summary.json --schema schemas/summary_schema.json &&
python check_preferences.py ${workspace} --rules P1,P2,P3 --target docs/report*.md
```

好处：
- 要求 agent 在单次任务内生成**多个相互关联的文件**，且各文件的数值须内部自洽
- 任一文件输出错误或数值不一致，整题失败，无法逐个蒙混
- 减少总题数，提高每题的信息密度

**设计要点**：多产物任务中，各文件之间须有**共享事实交叉引用**（M3 机制），check 脚本须跨文件校验这些共享值。

---

## 五、验证脚本的六类难度机制（M1–M6）

| 机制 | 说明 | 典型用法 |
|------|------|---------|
| **M1** | 解析 JSON/MD，验证派生计算值在严格容差内 | 时间差、比率、百分比 |
| **M2** | 要求 agent 明确引用两份冲突来源，说明哪个更可信及理由；check 验证结论方向 | 两版数据互相矛盾的场景 |
| **M3** | 多个输出文件交叉引用同一组事实；check 跨文件校验数值一致性 | report.md + summary.json 共享数字 |
| **M4** | JSON 须符合严格 schema（字段名、枚举值、类型完全匹配） | contradiction_tracker.json |
| **M5** | Agent 所写 Python 脚本被直接运行，check 验证 stdout JSON 字段精度 | L3 类脚本任务 |
| **M6** | 负向断言：check 验证 agent **未使用**错误来源数值作为结论 | 矛盾来源裁决后的二次验证 |

**L1 级检查（`test -f` + 单行 `grep`）不得单独成题**，只能作为 L2 command 的前置 `&&` 快速失败短路条件。

## 六、pref 字段：教学期 / 静默期两段制

**Phase 0–1（前两个 update 区间，约前 1/3 轮）**：pref 字段附实质提示，不计入得分

```jsonc
"pref": {
  "command": "python check_preferences.py ${workspace} --rules P1,P2",
  "expect_exit": 0,
  "feedback": {
    "correct": "",
    "incorrect": "格式提示：报告缺少 ## 级标题或 docs/ 目录缺少日期前缀文件"
  }
}
```

**Phase 2–4（后 2/3 轮）**：偏好逻辑迁入 `eval.command`，计入得分，agent 不再收到反馈：

```jsonc
"command": "python check_preferences.py ${workspace} --rules P1,P2,P3,P4,P5 --target docs/final_report_*.md"
```

## 七、MC 题的定位与数量

MC 只在三类位置保留，目标约 8 道 / 30 轮：

1. **场景开篇**（前 2–3 轮）：建立基线认知，引入核心矛盾
2. **每个 update 触发轮**：验证 agent 对新信息的整合
3. **最终综合**（最后 1–2 轮）：跨 update 综合结论或元认知

其余轮次均为 EC。MC 不作"过渡"或"填充"使用。

**update 密集区段处理**：若原始设计中两个 update 相邻（如 upd2@q7、upd3@q8），须重新设计 update_ids，在中间插入 3–4 道 EC，将间距拉开至 ≥4 轮。

## 八、造题前必建 Ground Truth 数值表

每个场景子计划开头须先整理：

| 事实 | 数值 | 来源文件 |
|------|------|---------|
| 下载时间戳 | `2026-09-25T14:22:17+08:00` | `cloud-storage-access-log.md` |
| 邮件发送时间戳 | `2026-09-25T15:03:44+08:00` | `email-attachment-audit.md` |
| 时间差（秒） | **2487** | 计算值 |
| 文件大小比率 | **2.875** | 计算值（2.3 / 0.8） |

**写完验证脚本后，必须将每个期望值回溯到表中对应行核实**。若某数值在 workspace 里找不到来源，则该题无解，须修改 workspace 或换题。

## 九、升级前后对比速查

| 维度 | v1（过简） | v2/v3（适当） |
|------|----------|-------------|
| EC 题本质 | 抄写 + 整理 | 推理 + 计算 + 多文件产出 |
| 验证粒度 | 关键词存在 / 文件非空 | ≥3 个具体数值 exact match + 负向断言 |
| L1 独立题 | 存在 | 禁止，只能作前置短路条件 |
| L3 任务 | 少量（每场景 1 题） | 扩展至 2–3 题，与路径 A 数值精度验证配合 |
| 同批次 EC 布局 | 分散多道小题 | 合并为多产物任务，`&&` 串联验证 |
| MC 数量 | ~7 道（偏多） | ~8 道，严格限于开篇/update触发/综合收尾 |
| pref 字段 | 随意附加 | 教学期有反馈（不计分）→ 静默期迁入 eval.command（计分） |
| Workspace 要求 | — | 尊重现有文件结构，出题前先 ls 核实 |

## 十、实操检查清单（每道 EC 题造完后对照）

- [ ] 题目是否包含真正的推理环节？（不能是纯搜索+粘贴）
- [ ] 验证是否检查 ≥3 个具体数值？（非关键词匹配）
- [ ] 每个期望值是否能在 workspace 文件中找到文档来源？
- [ ] 若存在矛盾来源，是否加入 M6 负向断言？
- [ ] 同批次 update 区间内的多道小题是否已合并为多产物任务？
- [ ] 多产物任务的各文件之间是否有共享事实交叉引用（M3）？
- [ ] L1 级检查是否只作为前置 `&&` 短路而非独立验证逻辑？
- [ ] 当前轮次的 pref 字段是否符合"教学期/静默期"分段规则？
- [ ] 与前后 MC/EC 的排列是否形成认知-行动交替节奏？
- [ ] 密集 update 区段相邻 update 之间是否有 ≥4 轮缓冲？
