## Overview

This data dictionary provides comprehensive documentation for all datasets used in the customer churn analysis project. It defines the structure, meaning, and relationships of data fields across multiple sources.

The primary goal of this documentation is to ensure data consistency, facilitate data understanding across teams, and serve as the authoritative reference for all data-related questions.

### Purpose of Documentation

- **Data Governance:** Establish clear ownership and definitions for all data fields
- **Cross-Team Alignment:** Enable marketing, product, and analytics teams to speak the same data language
- **Quality Assurance:** Document expected formats, constraints, and validation rules
- **Onboarding:** Help new team members understand our data landscape quickly
- **Compliance:** Support audit requirements and data lineage tracking

### Data Refresh Schedule

- **transactions_v1.csv**: Daily incremental updates at 2:00 AM UTC
- **customers.csv**: Daily full refresh at 3:00 AM UTC
- **products.csv**: Weekly updates on Sundays at 1:00 AM UTC

### Data Retention Policy

- **Raw transactional data**: Retained for 7 years
- **Aggregated metrics**: Retained for 10 years
- **Customer PII**: Subject to GDPR/CCPA deletion requests
- **Archived datasets**: Moved to cold storage after 2 years

## Dataset Schemas

### transactions_v1.csv

**Description:** Transaction-level data capturing all customer purchase events. Each row represents a single transaction with associated customer, product, and monetary information.

**Primary Key:** `transaction_id`
**Foreign Keys:** `customer_id` (→ customers.csv), `product_id` (→ products.csv)
**Row Count:** ~500,000 transactions (as of 2025-03-01)
**File Size:** ~45 MB
**Encoding:** UTF-8
**Delimiter:** Comma (,)
**Date Range:** 2023-01-01 to Present

#### Schema Definition

| Field | Type | Length | Nullable | Default | Description |
|-------|------|--------|----------|---------|-------------|
| transaction_id | string | 20 | No | - | Unique identifier for each transaction. Format: T{YYYYMMDD}{6-digit sequence} |
| customer_id | string | 15 | No | - | Foreign key linking to customers table. Format: C{8-digit number} |
| date | date | - | No | - | Date and time when transaction occurred (UTC timezone) |
| order_value | float | - | No | - | Total transaction amount in USD before taxes and discounts |
| product_id | string | 10 | Yes | NULL | Foreign key linking to products table. NULL for bundle purchases |
| quantity | integer | - | No | 1 | Number of units purchased in this transaction |
| payment_method | string | 20 | Yes | NULL | Payment type: credit_card, debit_card, paypal, gift_card |
| order_status | string | 15 | No | completed | Status: completed, cancelled, refunded, pending |
| discount_amount | float | - | Yes | 0.0 | Discount applied in USD |
| tax_amount | float | - | Yes | 0.0 | Tax amount in USD |
| shipping_cost | float | - | Yes | 0.0 | Shipping cost in USD |

#### Field Details - transactions_v1.csv

##### transaction_id
- **Purpose:** Uniquely identifies each transaction across all systems
- **Format:** T{YYYYMMDD}{6-digit sequence number}
  - Example: T20250312000001, T20250312000002
- **Generation:** Auto-generated by e-commerce platform at transaction time
- **Business Rules:**
  - Must be unique across all time periods
  - Sequence resets daily at midnight UTC
  - Cannot be reused even for cancelled/refunded orders
- **Data Quality:**
  - No nulls allowed
  - Must match regex pattern: `^T\d{14}$`
  - Duplicates indicate data pipeline errors
- **Usage Notes:**
  - Use for joining with order details and fulfillment systems
  - Include in all customer support queries
  - Required for refund processing

##### customer_id
- **Purpose:** Links transaction to customer record; enables customer-level analysis
- **Format:** C{8-digit zero-padded number}
  - Example: C00000001, C00123456
- **Foreign Key:** References customers.csv primary key
- **Business Rules:**
  - Must exist in customers table before transaction is recorded
  - Guest checkouts receive temporary customer_id (prefix: CG)
  - Account merges preserve original transaction history
- **Data Quality:**
  - No nulls (guest users get placeholder IDs)
  - Referential integrity enforced
  - Orphaned transactions (no matching customer) flagged for investigation
- **Privacy Considerations:**
  - Consider this PII - handle according to privacy policy
  - Mask in non-production environments
  - Include in data subject deletion requests
- **Usage Notes:**
  - Primary key for customer lifetime value calculations
  - Use for customer segmentation and cohort analysis
  - Critical for churn prediction models

##### date
- **Purpose:** Records when transaction occurred; enables time-series analysis
- **Format:** ISO 8601 date format (YYYY-MM-DD) or datetime (YYYY-MM-DD HH:MM:SS)
- **Timezone:** All timestamps stored in UTC; convert to local timezone as needed
- **Business Rules:**
  - Must be within last 5 years and not future-dated
  - Cancelled orders retain original transaction date
  - Refund transactions create new record with refund date
- **Data Quality:**
  - No nulls allowed
  - Anomaly detection for transactions outside business hours
  - Date validation: must be >= 2020-01-01 and <= current_date + 1 day
- **Known Issues:**
  - Prior to 2023-06-01: some dates stored in local timezone (PST)
  - Migration script applied to normalize all to UTC
  - See change history for details
- **Usage Notes:**
  - Primary field for time-series analysis and seasonality detection
  - Use for recency calculations in RFM analysis
  - Critical for churn definition (30-day inactivity threshold)
- **Related Fields:**
  - created_at (system timestamp, not exported)
  - updated_at (system timestamp, not exported)

##### order_value
- **Purpose:** Captures the monetary value of the transaction; primary revenue metric
- **Unit:** US Dollars (USD)
- **Precision:** 2 decimal places
- **Business Rules:**
  - Represents subtotal before taxes, shipping, and discounts
  - Minimum value: $0.01 (no zero-value orders allowed)
  - Maximum value: $100,000 (orders above require manual approval)
  - For refunds: stored as negative value in separate refund record
  - Bundle discounts reflected in this field (individual product prices not stored)
- **Calculation:**
  ```
  order_value = Σ(product_price × quantity) - promotional_discounts
  ```
- **Data Quality:**
  - No nulls allowed
  - Must be positive (negative values indicate data error)
  - Outliers (>$10,000) flagged for manual review
  - Statistical validation: 99th percentile = $850
- **Historical Context:**
  - Prior to 2024-01-01: field name was "transaction_amount"
  - Renamed to "order_value" for clarity (2024-01-15 migration)
  - **IMPORTANT:** Legacy scripts may still reference "transaction_amount"
  - **BUG ALERT:** Some data loading scripts incorrectly map to "amount" instead of "order_value"
- **Currency Conversion:**
  - International orders converted to USD at time of transaction
  - Exchange rates refreshed hourly from external API
  - Original currency and rate stored in separate audit table
- **Usage Notes:**
  - Primary metric for revenue analysis and customer LTV
  - Use for monetary value in RFM segmentation
  - Key predictor in churn models (spend behavior patterns)
- **Related Fields:**
  - gross_revenue = order_value + tax_amount + shipping_cost
  - net_revenue = order_value - discount_amount - refund_amount

##### product_id
- **Purpose:** Identifies which product was purchased; enables product-level analysis
- **Format:** P{6-digit number} or SKU format (varies by product category)
  - Example: P000123, P099999, SKU-ELEC-00145
- **Foreign Key:** References products.csv primary key
- **Business Rules:**
  - NULL for multi-product bundles (bundle_id stored separately)
  - Discontinued products retain their IDs for historical analysis
  - Product variants share base product_id with suffix (-R, -B, -L for color/size)
- **Data Quality:**
  - Approximately 5% NULL values (legitimate bundles)
  - Must exist in products catalog if not NULL
  - Orphaned product_ids (<1%) indicate catalog sync issues
- **Historical Context:**
  - Prior to 2023-03-01: product_id format was inconsistent
  - Standardization project completed 2023-04-15
  - Legacy mappings maintained in product_mapping.csv
- **Usage Notes:**
  - Join with products.csv for category and margin analysis
  - Use for product affinity and cross-sell analysis
  - Filter out NULLs when doing product-specific analysis
- **Related Fields:**
  - category_id (from products.csv join)
  - product_name (from products.csv join)

##### quantity
- **Purpose:** Number of units purchased in this transaction
- **Type:** Integer
- **Business Rules:**
  - Minimum: 1 (zero quantity not allowed)
  - Maximum: 999 (bulk orders above this go through B2B channel)
  - Most transactions (95%) have quantity = 1
  - Average quantity: 1.3 units
- **Data Quality:**
  - No nulls (default to 1 if missing)
  - Must be positive integer
  - Negative quantities indicate data error (refunds handled separately)
- **Usage Notes:**
  - Multiply by order_value for total revenue calculations
  - Use for inventory forecasting
  - High quantities may indicate business customers

### customers.csv

**Description:** Customer master data containing demographic information, account details, and churn labels. Each row represents a unique customer.

**Primary Key:** `customer_id`
**Row Count:** ~50,000 customers
**File Size:** ~8 MB
**Encoding:** UTF-8
**Update Frequency:** Daily

#### Schema Definition

| Field | Type | Length | Nullable | Default | Description |
|-------|------|--------|----------|---------|-------------|
| customer_id | string | 15 | No | - | Unique customer identifier (primary key) |
| signup_date | date | - | No | - | Date when customer account was created |
| email | string | 255 | Yes | NULL | Customer email address (PII) |
| first_name | string | 50 | Yes | NULL | Customer first name (PII) |
| last_name | string | 50 | Yes | NULL | Customer last name (PII) |
| age | integer | - | Yes | NULL | Customer age in years |
| gender | string | 10 | Yes | NULL | Gender: M, F, Other, PreferNotToSay |
| region | string | 50 | Yes | NULL | Geographic region: North, South, East, West, International |
| segment | string | 20 | Yes | Standard | Customer segment: Premium, Standard, Basic, VIP |
| is_churned | integer | - | No | 0 | Churn flag: 1=churned, 0=active |
| churn_date | date | - | Yes | NULL | Date when customer was marked as churned |
| last_active | date | - | Yes | NULL | Date of most recent activity (transaction or login) |
| lifetime_value | float | - | Yes | NULL | Calculated customer lifetime value in USD |
| account_status | string | 15 | No | active | Status: active, suspended, closed, dormant |

#### Field Details - customers.csv

##### customer_id
- **Purpose:** Unique identifier for each customer across all systems
- **Format:** C{8-digit zero-padded number}
  - Example: C00000001, C00012345
  - Guest users: CG{8-digit} (temporary, converted upon account creation)
- **Business Rules:**
  - Assigned sequentially at account creation
  - Never reused, even for deleted accounts
  - Account merges preserve both IDs with mapping table
- **Data Quality:**
  - Primary key - guaranteed unique
  - No nulls allowed
  - Format validation: ^C\d{8}$ or ^CG\d{8}$
- **Privacy:**
  - Considered PII - subject to data privacy regulations
  - Anonymized in test/development environments
  - Included in right-to-be-forgotten requests
- **Usage Notes:**
  - Join key for all customer-related analyses
  - Use for customer segmentation and cohort tracking
  - Required for churn prediction models

##### signup_date
- **Purpose:** Records when customer created their account
- **Format:** YYYY-MM-DD
- **Timezone:** UTC
- **Business Rules:**
  - Must be >= 2020-01-01 (platform launch date)
  - Cannot be future-dated
  - Imported accounts: signup_date = import_date
- **Data Quality:**
  - No nulls allowed
  - Validated against account creation logs
  - ~500 records have estimated signup_date (data migration artifacts)
- **Usage Notes:**
  - Calculate customer age/tenure
  - Cohort analysis by signup month
  - Time-to-first-purchase analysis

##### email
- **Purpose:** Primary contact method for customer communications
- **Format:** Standard email format (RFC 5322)
- **Business Rules:**
  - Must be unique per customer (one account per email)
  - Validated at signup (basic format check)
  - Unverified emails marked with flag (not in this export)
  - Bounced emails updated to NULL
- **Data Quality:**
  - ~3% NULL values (guests, email removed, opted out)
  - Format validation: basic regex check
  - Deduplication: case-insensitive uniqueness
- **Privacy:**
  - **HIGH SENSITIVITY PII**
  - Encrypted at rest
  - Masked in non-production environments
  - Included in data subject requests
- **Known Issues:**
  - Some legacy accounts have placeholder emails (noemail@domain.com)
  - These should be treated as NULL
- **Usage Notes:**
  - Use for email campaign targeting (with opt-in check)
  - Do NOT include in training datasets without anonymization
  - Consider domain analysis for B2B customer identification

##### age
- **Purpose:** Customer age for demographic analysis and segmentation
- **Unit:** Years
- **Business Rules:**
  - Valid range: 18-120 years
  - Self-reported at account creation (optional)
  - Not updated automatically (static snapshot)
  - Validation: must be 18+ for account creation
- **Data Quality:**
  - ~40% NULL values (not provided by customers)
  - Known data quality issues:
    - Some outliers (age > 100) likely data entry errors
    - Default value "99" used historically for "prefer not to say"
  - Statistical profile: mean=35, median=32, mode=28
- **Privacy:**
  - PII in some jurisdictions
  - Aggregated for reporting (age buckets recommended)
- **Usage Notes:**
  - Segment into age groups for analysis: 18-24, 25-34, 35-44, 45-54, 55+
  - High NULL rate limits usefulness
  - Consider imputation for predictive models
- **Related Fields:**
  - date_of_birth (not exported, stored internally)

##### gender
- **Purpose:** Demographic segmentation and personalization
- **Values:**
  - M: Male
  - F: Female
  - Other: Non-binary/other
  - PreferNotToSay: Declined to specify
  - NULL: Not provided
- **Business Rules:**
  - Self-reported, optional field
  - No validation beyond accepted values
  - May not reflect current identity (static field)
- **Data Quality:**
  - ~35% NULL values
  - Distribution: M (45%), F (48%), Other (2%), PreferNotToSay (5%)
  - Legacy data may have inconsistent capitalization
- **Privacy:**
  - PII in some jurisdictions
  - Handle with appropriate sensitivity
- **Usage Notes:**
  - Use for product recommendations and marketing segmentation
  - High NULL rate limits analytical value
  - Avoid making assumptions based on gender

##### region
- **Purpose:** Geographic segmentation for logistics and marketing
- **Values:**
  - North: Northern US states and Canada
  - South: Southern US states
  - East: Eastern US states
  - West: Western US states
  - International: All non-US/Canada
  - NULL: Not specified or unknown
- **Business Rules:**
  - Derived from shipping address (most recent)
  - Updated with each order (may change over time)
  - International includes all countries outside US/Canada
- **Data Quality:**
  - ~5% NULL values (no shipping address on file)
  - Distribution: North (22%), South (20%), East (25%), West (23%), International (10%)
- **Historical Context:**
  - Prior to 2023-01-01: only US regions tracked
  - International category added for global expansion
- **Usage Notes:**
  - Use for regional churn analysis
  - Consider timezone impacts on communication timing
  - Join with zip code data for more granular analysis

##### segment
- **Purpose:** Customer tier/value classification for targeted strategies
- **Values:**
  - VIP: Top 1% of customers by lifetime value (>$10,000)
  - Premium: High-value customers ($5,000-$10,000 LTV)
  - Standard: Regular customers ($1,000-$5,000 LTV)
  - Basic: Low-value/new customers (<$1,000 LTV)
  - NULL: Not yet classified (new accounts)
- **Business Rules:**
  - Recalculated monthly based on LTV
  - Can move up or down segments
  - Segments determine pricing, support tier, and promotions
  - New customers default to Standard after first purchase
- **Data Quality:**
  - ~8% NULL values (very new accounts, no purchases)
  - Distribution: VIP (1%), Premium (9%), Standard (65%), Basic (25%)
- **Business Logic:**
  ```
  if lifetime_value >= 10000: segment = 'VIP'
  elif lifetime_value >= 5000: segment = 'Premium'
  elif lifetime_value >= 1000: segment = 'Standard'
  else: segment = 'Basic'
  ```
- **Usage Notes:**
  - Use for churn risk stratification (VIP churn is critical)
  - Consider segment-specific retention strategies
  - Monitor segment transitions over time

##### is_churned
- **Purpose:** Binary flag indicating whether customer has churned
- **Type:** Integer (0 or 1)
- **Values:**
  - 0: Active customer
  - 1: Churned customer
- **Business Definition:**
  - Customer is churned if no transactions in last 30 days
  - AND no account activity (login, browse, cart) in last 45 days
  - Defined by Alex at 2025-03-03 kickoff meeting
- **Business Rules:**
  - Recalculated daily by automated job
  - Can transition from churned back to active (reactivation)
  - Dormant status precedes churn (warning stage)
- **Data Quality:**
  - No nulls (default to 0)
  - Current distribution: 85% active, 15% churned
  - Validated against activity logs
- **Historical Context:**
  - Definition changed 2024-09-01: was 60 days, now 30 days
  - Historical labels recalculated with new definition
  - Migration may cause discontinuity in trends
- **Usage Notes:**
  - **PRIMARY TARGET VARIABLE** for churn prediction models
  - Use for churn rate calculations and trend analysis
  - Compare with predicted churn scores for model validation
- **Related Fields:**
  - churn_date: when customer was first marked as churned
  - last_active: used to calculate churn status

##### churn_date
- **Purpose:** Records when customer first met churn criteria
- **Format:** YYYY-MM-DD
- **Business Rules:**
  - Set when is_churned transitions from 0 to 1
  - NULL for active customers
  - Not updated if customer reactivates (historical record)
  - Separate reactivation_date field tracks return
- **Data Quality:**
  - Consistent with is_churned flag
  - NULL when is_churned = 0
  - Must be >= signup_date
  - Must be <= last_active + 30 days
- **Usage Notes:**
  - Calculate time-to-churn from signup
  - Cohort analysis by churn month
  - Survival analysis modeling

##### last_active
- **Purpose:** Most recent date of customer activity
- **Format:** YYYY-MM-DD
- **Activity Types:**
  - Transaction (purchase, refund)
  - Login to account
  - Add to cart
  - Browse products
  - Contact support
- **Business Rules:**
  - Updated daily from activity logs
  - Includes all engagement types (not just purchases)
  - Used to calculate is_churned flag
- **Data Quality:**
  - No nulls for customers with any activity
  - Must be >= signup_date
  - Must be <= current_date
  - ~1% NULL (accounts created but never used)
- **Usage Notes:**
  - **CRITICAL for churn definition**
  - Calculate recency in RFM analysis
  - Days since last activity = current_date - last_active
  - Filter for active customers: last_active >= (current_date - 30 days)

##### lifetime_value
- **Purpose:** Total revenue generated by customer over their lifetime
- **Unit:** US Dollars (USD)
- **Calculation:**
  ```
  lifetime_value = SUM(order_value) - SUM(refunds) + projected_future_value
  ```
- **Business Rules:**
  - Includes all historical transactions
  - Excludes cancelled orders
  - Includes refunds as negative values
  - Projection uses cohort-based modeling
- **Data Quality:**
  - NULL for customers with no purchases
  - Minimum: $0.00
  - Maximum: $85,432 (outlier VIP customer)
  - Mean: $1,245, Median: $450
- **Recalculation:**
  - Updated monthly with new transaction data
  - Projection model retrained quarterly
- **Usage Notes:**
  - Primary metric for customer value segmentation
  - Use for CLV-based churn cost analysis
  - Compare actual vs. predicted LTV for model validation

## Data Types and Formats

### Date Formats

**Standard Format:** ISO 8601 (YYYY-MM-DD)

**Accepted Input Formats:**
- YYYY-MM-DD (preferred)
- MM/DD/YYYY (US format)
- DD/MM/YYYY (European format)
- YYYY-MM-DD HH:MM:SS (with time)

**Timezone:** All dates stored in UTC unless otherwise specified

### Numeric Formats

**Currency:**
- Format: Decimal with 2 decimal places
- Unit: US Dollars (USD)
- Example: 1234.56

**Percentages:**
- Format: Decimal (0.15 = 15%)
- Range: 0.0 to 1.0

**Integers:**
- Standard integer type
- No decimal places

### String Formats

**Email:**
- Format: RFC 5322 compliant
- Max length: 255 characters
- Case: Case-insensitive for uniqueness

**Customer ID:**
- Format: C{8 digits}
- Example: C00012345
- Pattern: ^C\d{8}$

## Data Quality Notes

### Known Issues

1. **Date Format Inconsistency (Resolved)**
   - Prior to 2023-06-01: mixed timezones in date field
   - Resolution: Migration script applied 2023-06-15
   - Impact: Minimal, affects <0.1% of records

2. **Email Deduplication**
   - ~500 customers share emails (family accounts)
   - Identified by duplicate email with different customer_id
   - Business decision: allowed for family sharing

3. **Missing Product IDs**
   - ~5% of transactions have NULL product_id
   - Reason: Bundle purchases (legitimate)
   - Workaround: Use bundle_id from separate table

4. **Age Outliers**
   - ~50 customers with age > 100
   - Likely data entry errors
   - Recommendation: Cap at 100 or exclude from analysis

### Data Quality Metrics

**transactions_v1.csv:**
- Completeness: 98.5%
- Accuracy: 99.2%
- Consistency: 99.8%
- Timeliness: 99.5%
- Overall Quality Score: 99.0%

**customers.csv:**
- Completeness: 85.3% (many optional fields)
- Accuracy: 97.5%
- Consistency: 98.9%
- Timeliness: 99.0%
- Overall Quality Score: 95.2%

### Validation Rules

**transactions_v1.csv:**
- transaction_id must be unique
- customer_id must exist in customers.csv
- order_value must be > 0
- date must be within valid range
- quantity must be >= 1

**customers.csv:**
- customer_id must be unique
- email must be unique (if not NULL)
- age must be between 18 and 120
- signup_date must be <= current_date
